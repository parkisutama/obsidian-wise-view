# Wise View - Copilot Custom Instructions

## Project Context & Identity

You are an expert TypeScript developer and Obsidian API specialist assisting in the development of **Wise View**, an Obsidian Community Plugin.
Wise View is a task-management-agnostic view enrichment plugin for Obsidian Bases. It provides richer data visualizations (Calendar, Kanban, Timeline, Gantt) without enforcing any specific task management workflow.

### Core Philosophy

- **Read-Only / Agnostic:** The plugin visualizes existing frontmatter data. It does NOT enforce schemas, manage task states, or create task hierarchies.
- **View Enrichment Only:** All logic related to task creation, quick capture, natural language date parsing, and recurrence algorithms has been intentionally stripped from the original codebase. Do not reintroduce them.

## Codebase Lineage & Architecture

This project is a heavily modified derivative work. When generating or modifying code, you must respect the existing architectural patterns derived from these sources:

1. **Base Architecture (Obsidian Planner):** - Origin: [SawyerRensel/Planner](https://github.com/anareaty/pretty-properties)
   - Retained: Calendar (FullCalendar), Kanban, Timeline (Markwhen) view implementations.
   - Removed: Task list view, item modals, quick capture, `chrono-node` (date parsing), `rrule` (recurrence), and blocking dependencies.
2. **Gantt Integration:**
   - Origin: Adapted from [lhassa8/obsidian-bases-gantt](https://github.com/lhassa8/obsidian-bases-gantt).
   - Library: Uses [frappe/gantt](https://github.com/frappe/gantt) for the rendering engine.
   - Objective: Seamlessly integrate Frappe Gantt into the Obsidian Bases API architecture inherited from the Planner codebase.
3. **Styling & Color Configuration:**
   - Integration: [anareaty/pretty-properties](https://github.com/anareaty/pretty-properties)
   - Objective: Utilize existing color configurations and CSS variables provided by `pretty-properties` to theme the views (especially Kanban cards and Gantt bars) rather than hardcoding new color logic.

## Technical Environment & Tooling

- **Language:** TypeScript (`"strict": true`).
- **Target:** Obsidian Desktop and Mobile (`isDesktopOnly: false`).
- **Bundler:** esbuild (Output: `main.js`).
- **Libraries:**
  - `obsidian` (Obsidian Plugin API)
  - `frappe-gantt` (Gantt chart visualization)
  - `fullcalendar` (Calendar visualization)
  - `markwhen` (Timeline visualization)

## AI Agent Directives

### 1. Feature Development (Do's)

- Focus strictly on mapping Obsidian frontmatter properties (e.g., `date_start`, `date_end`, `status`, `progress`) to the respective view engines (Frappe Gantt, FullCalendar).
- Ensure DOM elements created by third-party libraries (like Frappe Gantt SVG elements) are properly scoped within Obsidian's view containers (`this.containerEl`).
- Clean up all library instances, DOM listeners, and Obsidian events during view unmounting or plugin `onunload()`.
- Write clean, self-explanatory, and modular code. Avoid massive monolithic functions.

### 2. Strict Constraints (Don'ts)

- **NO Task Logic:** Do not write code to calculate task dependencies, auto-update statuses, or handle recurring tasks.
- **NO Extraneous Modals:** Do not implement quick capture modals or task editing interfaces unless explicitly instructed to bridge a native Obsidian Base feature.
- **NO Hardcoded Styles:** Do not inject hardcoded HEX/RGB colors for task categories. Read and apply the CSS classes/variables generated by the `pretty-properties` integration.
- **NO Generic Solutions:** When implementing Frappe Gantt, account for Obsidian's specific theme switching (light/dark mode) and Base API data lifecycle.

### 3. Gantt & Frappe-Specific Guidelines

- Map Obsidian Base items to the Frappe Gantt task object structure: `{ id, name, start, end, progress, dependencies }`.
- Ensure date formats passed to `frappe-gantt` are strictly parsed from the Obsidian frontmatter cache to avoid timezone or format mismatch errors.
- Hook into Frappe Gantt's event listeners (e.g., `on_click`, `on_date_change`) only to route data back to Obsidian's file modification API (`app.fileManager.processFrontMatter`), maintaining the single source of truth in the markdown files.

### 4. UI & Copy Guidelines

- Use sentence case for UI elements.
- Ensure all views remain responsive to Obsidian's pane resizing.
- Delegate UI styling to `styles.css` using Obsidian's standard CSS variables (`var(--background-primary)`, `var(--text-normal)`, etc.) combined with `pretty-properties` variables.
